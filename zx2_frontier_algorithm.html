The ZX2 frontier parser

Frontier mode turns LZ parsing from a local hunch into a shortest-path problem: pay literals, buy matches, and let the cheapest route through the segment win.

Core/misc/ZX2.hpp segmented dynamic programming BT-backed match search

Start with the real choice

ZX2 is an LZ compressor: its output is mostly literal bytes plus instructions to copy earlier bytes. The hard part is not decoding those instructions. It is deciding where they should be.

A literal is brutally honest: spend the byte. A match is a bet: pay a token, pay offset bits, maybe pay length extensions, then hope the copied span is long enough to amortize the overhead. That is why "found a repeat" is not the same as "should emit a match".

The parser is the part of the compressor that decides where the literals and matches go. The decoder is much simpler: it just follows the final token stream. All the frontier machinery exists only while compressing.

Literal runRaw bytes copied into the literal stream. Great when data is new, bad when it repeats.
MatchA length plus an offset. Great when the copied phrase is long enough to pay for the token and offset.
ParserThe optimizer that chooses a sequence of literal runs and matches before anything reaches the decoder.

The problem with a tempting match

Compression parsing is full of little traps. At byte 1000 you may find a decent repeated phrase and grab it. But the byte at 1001 might start a much better phrase.

The lazy parser handles the simplest version of that trap by looking one step ahead. ZX2 frontier mode goes further. It treats the next segment as a tiny map: every byte position is a place you can stand, and every literal or match is a move to another place.

The goal is not "take the best match now". The goal is "arrive at the end of this segment with the lowest encoded cost".

What "frontier" means here

Imagine a line sweeping left to right over the segment. Behind it, the parser has already priced the cheapest known way to stand at each byte boundary. At the line, it spends that known cost into the future.

A literal edge always exists: append one raw byte and advance by one. A match edge exists only when the BT match finder returns a repeated phrase whose encoded match cost beats the same bytes as literals.

The frontier is not a queue or a geometric object in the code. It is this compact table of best-known futures:

cost[rel]      best cost to reach byte rel
parent[rel]    previous position on the winning path
parentLen      emitted literal length or match length
parentOff      match offset, or zero for literals
One survivorEach byte position keeps one winning cost. Close losers are discarded, even if they carry a different literal-run phase.
Literal phaselitExtLeft remembers when the next literal byte will also force a length-extension byte.
Tracebackparent == UINT32_MAX means the winning stretch is literal text; any other parent means a match jump.

The price is counted in bits

The graph is useful only because every edge has a real price. ZX2 scores the same bytes the stream will later write: token bits, offset bits, literal payload bytes, and extension bytes.

For a literal edge, the usual cost is one payload byte. But literal runs have length fields, and at exact thresholds one more literal also buys another extension byte. That is why the frontier carries litExtLeft, a tiny countdown to the next literal-length toll.

For a match edge, the price is front-loaded: one token plus the raw offset bits, with extra length bytes only for long matches. In the source this is matchLenCost8; despite the suffix, the values are ordinary bit counts.

The accounting stays integer and mechanical: no floats, no entropy model, no statistical guess. The parser is only asking which tokenization is cheaper under ZX2's own format.

long repeatgood
very longgreat

A small scoring example

Suppose the offset takes 22 bits. A match also needs a token byte, so a short match starts with about 30 bits of overhead before length extensions. Replacing four literal bytes saves only 32 payload bits. That is barely worth anything, and after parser penalties it may be rejected.

Replacing twenty bytes is different. The literal version costs around 160 payload bits, while the match still pays the same offset and token overhead. The longer the copied phrase, the more the fixed overhead disappears into the noise.

This is the quiet reason the frontier parser does not simply prefer the longest match. It prefers the path whose total encoding price is smallest.

The whole algorithm in five moves

1

Cut the input into segments. The current code uses ZX2_FRONTIER_SEGMENT, default 32768 bytes. Each segment is one bounded shortest-path problem.

2

Seed the first cost. At the start of a segment, cost[0] = 0. The pending literal-run state is imported with litExtLeft[0], because a literal run can continue across segment boundaries.

3

Advance by one literal. From each reachable position, it offers a one-byte literal edge. Most literals cost 8 bits of payload; sometimes they also trigger a literal-length extension byte.

4

Walk the per-hash binary tree. findBestAndInsertBT inserts the current byte position into a tree of older strings with the same 4-byte hash, then returns the best useful repeat found during that insertion walk.

5

Trace back and emit. Once cost[segN] is finite, ZX2 follows parent[] backward, reverses the route, and feeds those literal runs and matches into BlockBuilder.

The code, translated

The actual loop in ZX2Compressor::compress is compact because the graph is acyclic. It does dynamic programming, literal-run phase tracking, BT match generation, and traceback in one forward sweep plus one backward pass.

for each segment:
    cost[0] = 0

    for each reachable position rel:
        relax literal edge rel -> rel + 1

        match = findBestAndInsertBT(position)
        if match exists:
            relax match edge rel -> rel + match.len

    require cost[segN] to be finite
    follow parent[] backward from segment end
    replay the recovered path into BlockBuilder

The word "relax" is the standard shortest-path operation: if a newly discovered route to a future position is cheaper than the old route, replace the old route and remember where it came from.

Why the graph has no cycles

Every edge moves forward. A literal moves from rel to rel + 1. A match moves from rel to rel + len. There is no backward edge, so the parser does not need Dijkstra, a heap, or repeated passes.

Why parent pointers are enough

Once cost[segN] is final, the cheapest path is encoded in reverse by parent[]. A missing parent marks a literal stretch; a real predecessor marks a match jump with parentLen and parentOff.

Why replay instead of emitting immediately

Immediate emission would make the parser greedy again. Frontier mode first decides the whole segment path, then emits. That separation is the entire trick.

The binary tree under the frontier

The frontier table decides which path is cheapest. The binary tree supplies the jump edges cheaply enough to make that decision practical.

For each 4-byte hash, ZX2 keeps a tree of previous positions whose next bytes begin the same way. Each node is an old source position. One child points toward lexicographically smaller suffixes; the other points toward larger suffixes.

At a new position, the compressor descends that tree. Each visited old position is both a possible match and a routing comparison for insertion. The walk measures the common prefix, scores the match if it can save bits, then splices the current position into the same tree.

The clever detail is the LCP bookkeeping: the walk carries lower bounds from the small and large sides, so repeated byte comparisons are skipped. This keeps the search closer to "compare what is still unknown" than "restart the string compare at byte zero".

hashes[h]The root old position for this 4-byte hash bucket.
btLinks[pos]The two child links for one old source position, stored in a window-sized ring.
small / largeThe binary-search-tree split over suffix order, stored as compact child deltas.

hashes[hash(current)] = current

    maybe keep node as the best match

    if old_suffix_after_len < current_suffix_after_len:

Segments are a speed bargain

A whole-block optimal parser would let an early decision compete with every later consequence. ZX2 deliberately refuses that bill.

Inside a segment, it finds the best path under its candidate model. At the boundary, it commits. That is why the arrays are sized by ZX2_FRONTIER_SEGMENT + 1 instead of by the whole block: memory stays bounded, cache behavior stays sane, and the parser has a fixed horizon.

One continuity rule is worth keeping: pending literals can cross a segment boundary. If one segment ends with raw bytes and the next begins with raw bytes, ZX2 can keep them as one literal run, preserving the literal-extension accounting.

Why this is not full optimal parsing

The word "frontier" can sound grander than the code. ZX2 frontier mode is intentionally narrow.

It keeps one winning state per byte position: one cost, one literal-extension countdown, one parent. It also takes one best scored match from the BT at each start position. It does not preserve a portfolio of near-tie literal phases, multiple lengths for the same offset, or several offsets whose prices are close.

That is the right compromise for this compressor: much less myopic than greedy lazy parsing, but still small enough to reason about and cheap enough to run as an encoder mode.

Greedy

Take the best local match, maybe with one lazy lookahead. Fast, but easily fooled.

Frontier

Keep the cheapest surviving state at each byte. Better local-global tradeoff, still compact.

Optimal

Keep many competing states and candidates. Stronger ratio, much more parser work.

The knobs and what they really control

ZX2 has two kinds of parameters. Some change the stream format or decoder assumptions; those are architectural. Others only change how hard the encoder searches before it writes the same kind of stream. Mixing those up makes the frontier parser harder to reason about than it is.

algorithmparser

Selects the parsing strategy. Lazy is the faster local parser. Frontier spends more encoder work to solve a segmented path problem. This changes which tokens are emitted, but the decoder still reads the same ZX2 stream family.

fast local choicesslower segment planning

ZX2_TOKEN_LIT_BITS / ZX2_TOKEN_MATCH_BITSformat

Controls how one token byte is split between literal-run length and match length. The current 3/5 split gives small literal lengths in the token and more match-length room before extensions. Changing this reshapes both the stream and the parser cost table.

more literal token roommore match token room

ZX2_MATCH_OFFSET_BITS_VALUEformat

Controls how many bits are available for match offsets: 16, 18, 20, or 22 in this source. More bits let a match refer farther back, improving ratio on long-range repetition. They also make every match offset more expensive and change the encoded stream layout.

smaller, faster offsetslarger history reach

windowKBsearch

Caps how much previous input the compressor is allowed to search. It is clipped by the encoded offset range, so a huge window cannot exceed what the offset bits can represent. Larger windows may find older matches, but they increase parser memory and search pressure.

less memorymore long-range matches

hashBitsmatch finder

Sets the number of hash buckets for the 4-byte match key. More buckets usually mean shorter collision chains and cleaner candidates, at the cost of a larger hash table. Too few buckets make unrelated positions compete in the same bucket.

smaller tablecleaner buckets

maxChainmatch finder

Limits how many old candidates the matcher examines at one position. This is the main compression-speed dial: higher values search harder and can improve ratio, but every extra probe happens inside a very hot encoder loop.

faster encodebetter candidate search

chainBitsstorage

Controls the ring size used for match links. Frontier mode's binary tree requires it to match the window bits, because tree child references are stored as deltas inside that window-sized ring. Lazy mode can use row storage instead.

smaller link ringfull-window BT links

ZX2_FRONTIER_SEGMENT_VALUEfrontier

Sets the byte length of one dynamic-programming problem. Larger segments give the parser a longer horizon and larger arrays; smaller segments commit sooner and spend less state. It changes encoder decisions, not the decoder format.

less parser statelonger planning horizon

ZX2_FRONTIER_GOOD_ENOUGH_LENGTH_VALUEfrontier

Allows frontier's BT search to stop scoring once it has found a match of a chosen length. Zero means disabled. It is a speed shortcut: useful only if the early stop does not discard matches that would alter the segment path.

exhaustive scoringearlier stop

ZX2_FRONTIER_SKIP_PERIOD_VALUEfrontier

Skips insertion on a regular cadence while still walking the existing BT for scoring. It reduces tree churn and memory traffic, but it also removes some future source positions from consideration.

denser BT historyless insertion work

ZX2_MIN_MATCH_LENGTH_VALUEformat

Defines the shortest legal match length. The current default is 4. Raising it removes short-match choices and simplifies some decisions, but it can lose real compression opportunities. It also changes the meaning of match length codes.

more short matchesfewer candidates

ZX2_MIN_OFFSETdecoder

Sets the closest allowed backward copy. The current value, 32, is deliberate: it makes short decoded match copies non-overlapping, which lets the decoder use simple 32-byte vector copies. Lower offsets might find a few extra matches but complicate fast copy behavior.

more nearby matchessimpler fast copies

ZX2_MIN_MATCH_GAIN8_VALUEparser bias

Requires a match to beat literals by extra cost before it is accepted by the lazy scorer. It is a conservative bias against marginal matches. Frontier mostly uses the direct parser save table, but the idea is the same: tiny wins are not always worth extra match ops.

accept marginal winsdemand clearer wins

ZX2_ZERO_LIT_MATCH_PENALTY8_VALUEparser bias

Adds a parser-only penalty when a match follows a previous match with no literal bytes between them. This discourages dense chains of tiny matches, often helping decode speed by reducing match op count while spending a little ratio.

more match chainingfewer zero-lit matches

ZX2_MATCH_INSERT_LIMIT_VALUEparser policy

Caps how many interior positions of an emitted match are inserted into match search. Zero means no artificial cap in this source. A cap can speed future searches and avoid noisy candidates, but it is a parser policy rather than a format rule.

full insertionless interior work

blockBytesblock shape

Controls how much input goes into one ZX2 block. Blocks reset some output organization and bound memory use. A small block limits long-range opportunities across block boundaries; a huge block lets the parser and output streams work over more data at once.

smaller blocksmore continuity

huffmansecond pass

Optionally Huffman-compresses the split streams after the LZ parse. It changes the block payload representation but not the frontier decision process itself. Useful when the extra encode/decode stage wins enough bytes to justify the work.

raw split streamsextra entropy pass

What the frontier buys, and what it spends

Frontier is best understood as a ratio mode, not a decoder idea. It spends compression CPU and encoder memory to choose better tokens, while leaving decompression on the same simple ZX2 stream.

ParserQuestion it asksStrengthCost
Greedy / lazy Is there a good match here, or maybe one byte later? Fast compression, simple state, good default behavior. Can miss choices whose benefit appears several bytes later.
Frontier What is the cheapest path through this segment? Finds non-local wins inside the segment using actual ZX2 token costs. Visits every byte, walks the BT, and stores traceback state.
Rich optimal Which of many competing states will age best? Potentially better ratio, especially with many near-tie candidates. More memory, more branches, more candidate generation, harder tuning.
Decoder stays clean

The compressed bytes are still ordinary ZX2 tokens. Frontier changes how the encoder chooses them, not how the decoder reads them.

Compressor pays

Every source byte becomes a parser state. Every state walks the binary-tree matcher. That is the expensive part.

The most important tiny detail

The parser does not emit while it is deciding. It records possible futures first. That separation is what turns a pile of match searches into a path problem.

The second most important detail

The cost model is tied to the actual stream format. Change token layout, offset width, or extension thresholds, and the parser's idea of a good path changes with them.

The useful mental model

ZX2 frontier mode is a small pathfinder living inside the compressor.

It walks a fixed-size segment. At every byte it asks: buy one more literal, or jump with the best match the BT can justify? It records the cheapest known way to stand at future byte boundaries. At the end, traceback turns those remembered choices into ordinary ZX2 tokens.

The decoder never sees any of this. The output is still literal lengths, match lengths, offsets, and extension bytes. Frontier is an encoder-only luxury: pay compression time now, usually get fewer bytes later.
