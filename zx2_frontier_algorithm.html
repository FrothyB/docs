The parser's bargain

ZX2 is an LZ compressor: its output is a sequence of literal bytes and backward references into text already seen. The decoder follows those instructions mechanically. The parser has the harder problem: deciding which instructions should exist.

A literal is the direct statement of a byte. A match is an abbreviation: pay for a token, an offset, and sometimes extra length, then recover many bytes by reference. A repeat in the input is only an opportunity; it becomes a match only when the abbreviation is cheaper than the words it replaces.

btopt is the parser mode that treats this as an optimization problem. Over a bounded segment, every byte boundary is a vertex; every legal literal or match is a directed edge; every edge has the exact number of bits it will cost in the ZX2 stream. The parse is the minimum-cost path.

For a cursory read, that is the whole idea: btopt spends encoder work to buy non-local decisions while the decoder stays unchanged. The rest of the page unpacks what "minimum", "bounded", and "legal edge" mean in the actual implementation.

The trap one byte later

Parsing is full of near misses. At byte 1000 a decent repeated phrase may begin; at byte 1001, a better one. A local parser can spend its one chance too soon.

The lazy parser handles the simplest form of that problem by peeking one byte ahead. btopt gives itself a longer horizon. It treats the next segment as a map: every byte boundary is a position, and every literal or match is a forward edge.

The question stops being "which match looks best here?" and becomes "which route reaches the segment end at the lowest encoded cost?"

btopt as a frontier

Imagine a survey line moving left to right. Behind it, every byte boundary already has a best known cost. At the line, btopt extends that cost forward: one literal step, or one match edge.

The literal step is always available: append one raw byte and advance. A match edge appears only when the BT match finder returns a repeated phrase whose encoded cost beats writing those bytes literally.

The frontier is not a grand object in the code. It is a compact dynamic-programming table of best-known futures:

The accounting is in bits

The map only works because every edge has a real price. ZX2 charges exactly what the stream will later write: token bits, offset bits, literal payload bytes, and extension bytes.

A literal edge usually costs one payload byte. But literal runs have length fields, and at exact thresholds one more literal also requires another extension byte. That is why btopt carries litExtLeft, a small countdown to the next literal-length boundary.

A match edge is front-loaded: one token plus raw offset bits, with extra length bytes only for long matches. In the source this is matchLenCost8; despite the suffix, the values are ordinary bit counts.

The accounting stays integer and mechanical: no floats, no entropy estimate, no probabilistic model. The parser is only asking which tokenization is cheaper under ZX2's own format.

A small cost dispute

Suppose the offset takes 22 bits. A match also needs a token byte, so a short match starts with about 30 bits of overhead before length extensions. Replacing four literal bytes saves only 32 payload bits. That is barely a win, and parser penalties can erase it.

Replacing twenty bytes is different. The literal version costs around 160 payload bits, while the match still pays the same offset and token overhead. The longer the copied phrase, the less that fixed overhead matters.

The calculator makes the threshold visible. In this simplified range there are no length-extension bytes, so the match price is just 8 + offsetBits; literals cost 8 * len.

This is why btopt does not simply prefer the longest match. It prefers the cheapest complete route.

The model here deliberately uses the same simple costs as the diagrams: each literal byte costs 8 bits, and each match costs one token byte plus the selected offset width. It is not a byte-for-byte ZX2 encoder, but it preserves the mathematical issue: local matches can be globally expensive.

btopt wins by delaying commitment. Lazy chooses as it walks. btopt fills the cost table first, then reads the path backward. That single pause is the raison d'etre of the algorithm.

The whole algorithm in five moves

1

Cut the input into segments. The current code uses ZX2_FRONTIER_SEGMENT, default 32768 bytes. Each segment becomes one bounded btopt route-planning problem.

2

Seed the first cost. At the start of a segment, cost[0] = 0. The pending literal-run state rides in through litExtLeft[0], because a literal run may cross the border.

3

Offer the literal step. From each reachable position, the parser can always append one more literal byte. Usually that is 8 bits; occasionally the length field adds another byte too.

4

Ask the tree for matches. findBestAndInsertBT inserts the current byte position into a tree of older strings with the same 4-byte hash, then returns the best useful repeat found along the way.

5

Trace back and emit. Once cost[segN] is finite, btopt follows parent[] backward, reverses the route, and feeds those literal runs and matches into BlockBuilder.

The code, translated

The btopt loop in ZX2Compressor::compress is short because the graph has no cycles. It does dynamic programming, literal-run phase tracking, BT match generation, and traceback in one forward sweep plus one backward pass.

for each segment:
    cost[0] = 0

    for each reachable position rel:
        relax literal edge rel -> rel + 1

        match = findBestAndInsertBT(position)
        if match exists:
            relax match edge rel -> rel + match.len

    require cost[segN] to be finite
    follow parent[] backward from segment end
    replay the recovered path into BlockBuilder

"Relax" is the standard shortest-path word for a simple act: if a newly found route to a future byte is cheaper, replace the old cost and remember where the new one came from.

Why the graph has no cycles

Every edge moves forward. A literal goes from rel to rel + 1. A match goes from rel to rel + len. Nothing points backward, so the parser does not need Dijkstra, a heap, or repeated passes.

Why parent pointers are enough

Once cost[segN] is final, the route is written backward in parent[]. A missing parent marks a literal stretch; a real predecessor marks a match jump with parentLen and parentOff.

Why replay instead of emitting immediately

Immediate emission would put the parser back in the local-choice trap. btopt first decides the whole segment route, then emits. That pause is the trick.

The tree that finds matches

The btopt cost table decides which route is cheapest. The binary tree supplies candidate match edges quickly enough for that decision to be worth making.

For each 4-byte hash, ZX2 keeps a tree of previous positions whose next bytes start the same way. Each node is an old source position. One child points toward lexicographically smaller suffixes; the other points toward larger suffixes.

At a new position, the compressor descends that tree. Every visited old position is both a candidate match and a comparison point for insertion. The walk measures the common prefix, scores the match if it saves bits, then splices the current position into the same tree.

The important refinement is the LCP bookkeeping: the walk carries lower bounds from the small and large sides, so it does not keep rereading bytes it already knows are equal. The search spends its attention on the first unknown byte, where the ordering decision actually lives.

A simpler hash-map match finder stores recent positions in a bucket and scans them as an unordered row. That is attractive for lazy parsing: low ceremony, predictable storage, good enough when you need one local answer. btopt asks a harder question at every byte, so candidate quality matters more. The tree spends a little structure to make the search resemble a suffix-order probe rather than a blind collision scan.

Binary tree vs hash row

Both designs begin with the same 4-byte hash. The difference is what happens after the bucket is found.

A row hash table says: here are the last k positions with this prefix; compare them one by one. It is compact and often fastest for a lazy parser because the parser only needs to know whether the current position has a good enough match.

A binary tree says: among positions with this prefix, maintain an order by the full suffix. Searching for the current suffix then walks a changing set of near neighbors. Near neighbors in suffix order are exactly where long common prefixes live, so the work is better aimed at candidates that can matter.

The tree is not magic and not free. It has child links, updates on insertion, branchy comparisons, and a bounded maxChain. Its justification is that btopt multiplies match search by every byte in the segment. A weak candidate generator poisons the dynamic program with missing edges; a better one gives the optimizer enough graph to optimize.

Segments bound the horizon

A whole-block optimal parser would let an early decision compete with every later consequence. btopt chooses a smaller, more tractable horizon.

Inside a segment, it finds the best route under its candidate model. At the boundary, it commits. That is why the arrays are sized by ZX2_FRONTIER_SEGMENT + 1 instead of by the whole block: memory stays bounded, cache behavior stays sane, and the parser has a fixed horizon.

One border rule is worth keeping: pending literals can cross a segment boundary. If one segment ends with raw bytes and the next begins with raw bytes, ZX2 can keep them as one literal run, preserving the literal-extension accounting.

Optimal, after modeling

"Optimal" in btopt has a precise scope. Once the segment length, state representation, legal edges, and candidate matches are fixed, the dynamic program finds the cheapest path through that graph. It is not claiming the globally shortest possible ZX2 stream among every conceivable match candidate in the whole block.

The implementation keeps one winning state per byte position: one cost, one literal-extension countdown, one parent. It also takes one best scored match from the BT at each start position. It does not keep a portfolio of near-tie literal phases, multiple lengths for the same offset, or several offsets whose prices are close.

That restraint is the point and the raison d'etre: far less myopic than greedy lazy parsing, but still small enough to reason about and cheap enough to run as an encoder mode.

What btopt buys

btopt is a ratio mode, not a decoder idea. It spends compression CPU and encoder memory to choose better tokens, while decompression stays on the same simple ZX2 stream.

ParserQuestion it asksStrengthCost
Greedy / lazy Is there a good match here, or maybe one byte later? Fast compression, simple state, good default behavior. Can miss choices whose benefit appears several bytes later.
btopt What is the cheapest route through this segment? Finds non-local wins inside the segment using actual ZX2 token costs. Visits every byte, walks the BT, and stores traceback state.
Rich optimal Which competing states deserve to survive? Potentially better ratio, especially with many near-tie candidates. More memory, more branches, more candidate generation, harder tuning.
Decoder stays clean

The compressed bytes are still ordinary ZX2 tokens. btopt changes the encoder's choices, not the decoder's job.

Compressor pays

Every source byte becomes a parser state. Every state walks the binary-tree matcher. That is where the extra encoder work goes.

The most important detail

The parser does not emit while it is deciding. It records possible futures first. That pause turns a pile of match searches into a path problem.

The second most important detail

The cost model is tied to the actual stream format. Change token layout, offset width, or extension thresholds, and the parser's preferences change with them.

The working model

btopt is a small route planner inside the ZX2 compressor.

It walks a fixed-size segment. At every byte it asks: emit one more literal, or jump with the best match the BT can justify? It records the cheapest known way to reach future byte boundaries. At the end, traceback turns those remembered choices into ordinary ZX2 tokens.

The decoder never sees the planning. The output is still literal lengths, match lengths, offsets, and extension bytes. btopt is encoder-only luxury: pay compression time now, usually get fewer bytes later.
